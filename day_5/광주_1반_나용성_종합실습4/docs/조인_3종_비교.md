# 3가지 조인(Join) 방식 비교 — Nested Loop / Hash / Merge

> SKALA day_5 종합실습 4 · PostgreSQL 18.4 · schema `ecom`
> 실행 스크립트: `sql/03_join_3types.sql`

PostgreSQL 옵티마이저는 두 테이블을 조인할 때 아래 3가지 물리 조인 알고리즘 중
**비용(cost)이 가장 낮다고 추정되는 것**을 자동으로 고른다. 개발자가 직접 고르는 게
아니라, 통계(행 수·분포)·인덱스·정렬 상태를 보고 옵티마이저가 결정한다.

---

## 한눈에 보는 비교표

| 구분 | Nested Loop | Hash Join | Merge Join |
|------|-------------|-----------|------------|
| 동작 원리 | outer 각 행마다 inner를 조회(이중 루프) | 작은 쪽으로 해시테이블 빌드 → 큰 쪽을 프로브 | 양쪽을 조인키로 정렬 후 지퍼처럼 병합 |
| 유리한 조건 | outer가 소량 + inner에 조인키 **인덱스** | **대량** 등가조인(=), 인덱스 부적합 | 양쪽 입력이 이미 조인키로 **정렬**됨 |
| 조인 조건 | `=`, 범위, 부등호 모두 가능 | 등가(`=`)만 | 등가(`=`) 및 범위 |
| 사전 준비 | inner 인덱스 | 해시테이블(메모리) | 양쪽 정렬(Sort 비용) |
| 메모리 | 적음 | `work_mem`(해시) | 정렬용 `work_mem` |
| 약점 | outer가 크면 반복 폭발(O(N×M)) | 해시가 `work_mem` 초과 시 배치 분할(디스크) | 정렬 비용이 큼 |
| 대표 사례(ecom) | 특정 고객 1명의 주문 조회 | 실매출 주문 전체 × 주문상세 집계 | orders × payments (PK 정렬 스트림) |

---

## (1) Nested Loop Join — "outer 소량 + inner 인덱스"

```sql
SELECT o.order_id, o.order_ts, oi.product_id, oi.line_total
FROM orders o
JOIN order_items oi ON oi.order_id = o.order_id
WHERE o.customer_id = 7;     -- outer(orders)가 15행뿐
```

실측 실행계획(요약):

```
Nested Loop  (actual time=0.024..0.071 rows=36 loops=1)
  ->  Bitmap Heap Scan on orders o   (rows=15)        -- outer: 고객 7의 주문 15건
        ->  Bitmap Index Scan on idx_orders_customer_ts (customer_id = 7)
  ->  Index Only Scan on order_items oi (rows=2.4 loops=15)  -- inner를 15번 인덱스 조회
```

- **핵심**: `loops=15` — outer 15행 각각에 대해 inner를 인덱스로 콕 집어 조회.
- outer 결과가 적을 때 가장 빠르다(포인트 조회). outer가 커지면 반복이 폭발해 불리.

## (2) Hash Join — "대량 등가조인의 기본값"

```sql
SELECT count(*), SUM(oi.line_total)
FROM orders o
JOIN order_items oi ON oi.order_id = o.order_id
WHERE o.order_status IN ('paid','shipped','delivered');
```

실측 실행계획(요약):

```
Hash Join  (actual time=1.7..6.7 rows=19858)
  Hash Cond: (oi.order_id = o.order_id)
  ->  Seq Scan on order_items oi (rows=26083)        -- 큰 쪽을 훑으며 프로브
  ->  Hash (rows=7286)                                -- 작은 쪽(orders)으로 빌드
        ->  Seq Scan on orders o (Filter: status IN ...)
```

- **핵심**: 작은 입력(orders 7,286행)으로 해시테이블을 만들고, 큰 입력(order_items
  26,083행)을 한 번 훑으며 매칭. 인덱스 없이도 대량 등가조인을 효율적으로 처리.
- 집계·리포트성 쿼리 대부분이 여기에 해당(GMV, 카테고리 매출 등).
- 등가(`=`) 조인에서만 가능. 해시가 `work_mem`을 넘으면 배치로 나뉘어 디스크 사용.

## (3) Merge Join — "양쪽 정렬 입력의 병합"

```sql
-- (학습용) 소규모에선 Hash를 선호하므로 다른 전략을 막아 유도
SET enable_hashjoin = off;
SET enable_nestloop = off;
SELECT o.order_id, o.order_status, p.method, p.amount
FROM orders o
JOIN payments p ON p.order_id = o.order_id;
SET enable_hashjoin = on;   -- 원복
SET enable_nestloop = on;
```

실측 실행계획(요약):

```
Merge Join  (actual time=0.016..2.588 rows=7940)
  Merge Cond: (o.order_id = p.order_id)
  ->  Index Only Scan using orders_pkey on orders o   (rows=9560, 정렬 스트림)
  ->  Index Scan using idx_payments_order on payments p (rows=7940, 정렬 스트림)
```

- **핵심**: 양쪽 입력이 이미 `order_id` 순서로 정렬돼 있어(PK/인덱스), 두 스트림을
  지퍼처럼 맞물려 한 번에 병합. 별도 정렬 없이 진행 가능.
- 양쪽이 대량이고 정렬돼 있거나, 정렬 비용을 상쇄할 만큼 결과가 클 때 유리.
- 실무에선 옵티마이저가 소규모 데이터에서 Hash Join을 더 싸게 보는 경우가 많아,
  Merge Join은 대량·정렬 상황에서 자연스럽게 선택된다.

---

## 정리 — 언제 무엇이 선택되나

- **Nested Loop**: "조금 뽑아서(outer) 인덱스로 콕콕(inner)" → 포인트/소량 조회.
- **Hash Join**: "많이 뽑아서 한 번에 매칭" → 대량 등가조인·집계의 기본.
- **Merge Join**: "이미 줄 세워져 있으니 나란히 합치기" → 대량 정렬 입력.

> 튜닝 관점: 조인 알고리즘은 직접 지정하기보다 **통계를 최신화(ANALYZE)** 하고
> **적절한 인덱스**를 제공해 옵티마이저가 올바른 선택을 하도록 돕는 것이 정석이다.
> `SET enable_*` 플래그는 학습·디버깅용이며 운영 쿼리에 상시 사용하지 않는다.
