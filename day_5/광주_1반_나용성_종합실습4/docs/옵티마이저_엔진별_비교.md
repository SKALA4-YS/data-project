# (Option) DBMS 엔진별 옵티마이저 조사

> SKALA day_5 종합실습 4 · 실행계획 비교 & 성능 개선 — 심화 조사
> PostgreSQL / MySQL(InnoDB) / Oracle / SQL Server

## 0. 공통 개념 — 옵티마이저란

SQL은 "무엇을(what)" 구할지만 선언하고, "어떻게(how)" 가져올지는 **옵티마이저**가 정한다.
현대 상용/오픈소스 DB는 대부분 **CBO(Cost-Based Optimizer, 비용 기반)** 를 사용한다.
통계(행 수·값 분포·NDV 등)를 바탕으로 여러 실행계획 후보의 비용을 추정해 최저 비용 계획을 고른다.

- 조인 알고리즘(Nested Loop / Hash / Merge), 조인 순서, 스캔 방식(Seq/Index/Bitmap),
  집계 방식(Hash/Sort) 등이 옵티마이저의 선택 대상이다.
- 좋은 계획의 전제는 **정확한 통계**. 통계가 낡으면 오추정 → 나쁜 계획이 나온다.

## 1. 엔진별 특징 비교표

| 항목 | PostgreSQL | MySQL (InnoDB) | Oracle | SQL Server |
|------|------------|----------------|--------|------------|
| 옵티마이저 유형 | CBO | CBO | CBO (과거 RBO→현재 CBO) | CBO |
| 지원 조인 알고리즘 | Nested Loop / Hash / Merge | Nested Loop 계열(BNL, Hash Join은 8.0.18+) | Nested Loop / Hash / Sort-Merge | Nested Loop / Hash / Merge |
| 실행계획 확인 | `EXPLAIN (ANALYZE, BUFFERS)` | `EXPLAIN` / `EXPLAIN ANALYZE`(8.0.18+) | `EXPLAIN PLAN` + `DBMS_XPLAN`, AutoTrace | `SET SHOWPLAN_*`, 실제 실행계획(SSMS) |
| 통계 수집 | `ANALYZE`(autovacuum이 자동) | `ANALYZE TABLE`(InnoDB 자동 샘플링) | `DBMS_STATS`(자동 통계 잡) | `UPDATE STATISTICS`(auto update stats) |
| 힌트(Hint) | 공식 힌트 없음(플래그·`pg_hint_plan` 확장) | 옵티마이저 힌트 `/*+ ... */` | 강력한 힌트 문법 `/*+ ... */` | 쿼리 힌트 `OPTION (...)`, 조인 힌트 |
| 플랜 안정성 기능 | (기본 없음) | — | SQL Plan Baseline / SQL Profile | Query Store, Plan Guide |
| 계획 캐시 | 세션 prepared plan | — | Shared Pool(커서 공유) | Plan Cache |
| 대표 특이점 | Bitmap Index Scan, 부분/표현식 인덱스, 풍부한 확장 | 인덱스 의존도 큼(예전엔 Hash Join 없음), 클러스터형 PK | 매우 성숙한 CBO, 적응형 계획(Adaptive Plans) | 적응형 조인(Adaptive Joins), 컬럼스토어 |

## 2. 엔진별 상세

### PostgreSQL
- 3대 조인(NL/Hash/Merge)을 모두 지원하고, **Bitmap Index Scan**(여러 인덱스/조건을
  비트맵으로 결합) 이 특징. 이번 실습에서 기간 필터 튜닝 시 Seq Scan → Bitmap Heap Scan
  전환이 대표 사례.
- **부분 인덱스(partial)·표현식 인덱스·커버링(INCLUDE)** 등 유연한 인덱스가 강점.
- 공식적인 SQL 힌트가 없다. 대신 `enable_seqscan` 같은 플래그(디버깅용)나
  `pg_hint_plan` 확장을 쓴다. 정석은 통계·인덱스로 옵티마이저를 돕는 것.
- `autovacuum`이 통계를 자동 갱신. 대량 적재 후엔 수동 `ANALYZE` 권장.

### MySQL (InnoDB)
- 전통적으로 **Nested Loop 중심**이었고, Hash Join은 8.0.18부터 지원(등가조인 한정).
- 인덱스 설계 의존도가 특히 크다(적절한 인덱스가 없으면 성능 급락).
- 테이블이 **클러스터형 인덱스(PK 기준 물리 정렬)** 로 저장 → PK 조회에 유리,
  세컨더리 인덱스는 PK를 포함(커버링 아니면 PK로 재조회).
- `EXPLAIN ANALYZE`(8.0.18+)로 실측 계획 확인. 옵티마이저 힌트 `/*+ JOIN_ORDER(...) */` 등.

### Oracle
- 가장 성숙한 CBO 중 하나. NL/Hash/Sort-Merge 모두 지원.
- **SQL Plan Baseline / SQL Profile / SPM** 로 계획을 고정·관리(플랜 회귀 방지)가 강력.
- **Adaptive Plans / Adaptive Statistics**: 실행 중 실제 행수를 보고 계획을 조정.
- 힌트 문법이 풍부(`/*+ USE_HASH(t) INDEX(t idx) LEADING(...) */`).
- `DBMS_STATS`로 통계 관리, `DBMS_XPLAN.DISPLAY_CURSOR`로 실제 계획 확인.

### SQL Server
- CBO + **Query Store**(계획 이력/회귀 탐지/강제)와 **Plan Guide** 로 계획 안정성 관리.
- **Adaptive Joins**(런타임에 Hash↔NL 전환), **컬럼스토어 인덱스**(분석 워크로드) 지원.
- 쿼리 힌트 `OPTION (HASH JOIN)`, `OPTION (RECOMPILE)` 등, 조인 힌트 제공.
- 자동 통계 갱신(auto update statistics), `UPDATE STATISTICS`로 수동 갱신.

## 3. 실무 시사점 (엔진 공통)

1. **통계가 핵심**: 어떤 엔진이든 통계가 낡으면 오추정 → 나쁜 계획. 대량 변경 후 통계 갱신.
2. **인덱스는 만능이 아니다**: 이번 실습에서 보듯 소규모/고선택도(대부분 통과) 쿼리는
   Full/Seq Scan이 최적. 옵티마이저의 "인덱스 미사용"이 정상일 수 있다.
3. **힌트는 최후 수단**: Oracle/MySQL/SQL Server는 힌트가 강력하지만, 남용하면 데이터
   변화에 취약. 먼저 통계·인덱스·쿼리 재작성으로 해결한다.
4. **계획 안정성**: 운영에서 계획 회귀가 문제라면 Oracle SPM, SQL Server Query Store 같은
   기능으로 검증된 계획을 고정하는 접근이 있다(PostgreSQL은 확장/재작성으로 대응).

> 요약: 네 엔진 모두 비용 기반 옵티마이저로 수렴했으나, **힌트/플랜관리 성숙도**는
> Oracle·SQL Server가 앞서고, **인덱스 유연성(부분/표현식/Bitmap)** 은 PostgreSQL이,
> MySQL은 클러스터형 인덱스와 인덱스 설계 의존이 특징이다.
